LOCAL TESTNET MIGRATION TEST — INSTRUCTIONS
=============================================

OVERVIEW
--------
This test creates a private two-node testnet with compressed hardfork heights
and automatically mines through all forks. Mining happens in two phases:

  Phase 1 (blocks 1-59):  Mine to miner wallets — earns coinbase rewards.
  Phase 2 (blocks 60+):   Mine to burn address — HF11 requires all coinbase
                           to go to the burn address. These blocks advance the
                           chain so the phase 1 coinbase rewards unlock.

After mining completes, the miner wallets have spendable ETN and the chain
is past the migration fork, ready for GUI migration tool testing.

Validator block signatures are mocked in the source code — sign_block() writes
a dummy 64-byte signature and verify_block_signature() always returns true.
No --fallback-to-pow flag is needed.


PREREQUISITES
-------------
1. Apply the source code changes listed in "FILES MODIFIED" below.
   A ready-to-apply patch is included:

     cd ~/electroneum
     git apply gui/migration-tool/toy-blockchain-testing-files/testnet-toy-hardforks.patch

   If the patch doesn't apply cleanly, each change is listed inline below
   so you can make the edits manually.

2. Rebuild electroneumd and electroneum-wallet-rpc:

     cmake -DCMAKE_BUILD_TYPE=Release ../../../..
     make -j$(sysctl -n hw.ncpu) electroneumd electroneum-wallet-rpc electroneum-wallet-cli

3. Ensure the rebuilt binaries are in:
     ~/electroneum/build/Darwin/gui-migration-app/release/bin/


TOY HARDFORK TABLE (testnet)
----------------------------
  HF1  @ height 1    (genesis)
  HF6  @ height 10
  HF7  @ height 20
  HF8  @ height 30   (public tx / mocked validator signatures)
  HF9  @ height 40
  HF10 @ height 50
  HF11 @ height 60   (migration fork — coinbase must go to burn address)


WHY TWO MINING PHASES?
----------------------
At HF11, validate_miner_transaction() enforces that all coinbase outputs go
to the burn address (etnkCys4uGh...). This means after height 60 you cannot
mine to your own wallet. But you still need blocks to be produced after
height 60 so that the coinbase rewards earned in phase 1 (blocks 1-59) get
enough confirmations to become spendable (5 blocks after HF8).

The script handles both phases automatically.


MOCKED VALIDATOR SIGNATURES
----------------------------
From HF8 onward, blocks require a validator signature. Rather than using the
real validator infrastructure (which requires keys from Electroneum Ltd), the
following functions are mocked in blockchain.cpp:

  sign_block():            Fills b.signature with 64 dummy bytes, varying by
                           block height so consecutive blocks differ.
  verify_block_signature(): Always returns true. Sets b.signatory from the
                           signature bytes.
  Validator list checks:   Skipped entirely (no isEnabled/isValid calls).
  Sequential signatory:    Check removed (mocked signatories vary by height).

This allows the miner to produce valid signed blocks without real validator keys.

STEP-BY-STEP
=============

STEP 1 — Run the test script
-----------------------------
The script starts both daemons and handles all mining automatically:

  cd ~/electroneum/gui/migration-tool/toy-blockchain-testing-files
  ./migration-test.sh

It will:
  1. Launch Daemon 1 and Daemon 2 (detached, peered exclusively)
  2. Mine to miner wallets up to height 59 (phase 1)
  3. Stop mining, switch to burn address
  4. Mine to burn address up to height 150 (phase 2)
  5. Stop mining and print status

With --fixed-difficulty 1, the whole process takes seconds.


STEP 2 — Get wallet keys
-------------------------
The GUI migration tool needs the wallet address, spend key, and view key.
Open a wallet in the CLI to retrieve them:

  ~/electroneum/build/Darwin/gui-migration-app/release/bin/electroneum-wallet-cli \
    --testnet --wallet-file ~/electroneum/gui/migration-tool/toy-blockchain-testing-files/test-wallets/miner1 --password "" --offline

Then in the wallet CLI:
  spendkey        (reveals secret spend key — copy it)
  viewkey         (reveals secret view key — copy it)
  exit


STEP 3 — Run the GUI migration tool
------------------------------------
The tool has been pre-configured to use the local testnet daemon
(127.0.0.1:34568) with SSL disabled and the --testnet flag on wallet-rpc.

  cd ~/electroneum/gui/migration-tool
  npm run dev

In the GUI:
  1. Enter the miner wallet address and its spend key + view key.
  2. The GUI will sync the wallet against the local daemon.
  3. After sync, it will show the migration status and derived SC address.


STEP 4 — Clean up
------------------
Stop both daemons:

  cd ~/electroneum/gui/migration-tool/toy-blockchain-testing-files
  ./migration-test.sh stop

Remove blockchain data (to start fresh next time):

  ./migration-test.sh clean

TROUBLESHOOTING
===============

"Chain stuck at height 60"
  - This means mining is still targeting a miner wallet address. After HF11,
    coinbase must go to the burn address. Stop mining and restart with the
    burn address (see Phase 2 above).
  - If using the script, this is handled automatically.

"Daemon not responding"
  - Check if electroneumd processes are running: ps aux | grep electroneumd
  - Check daemon logs in testnet-data-1/testnet/electroneum.log

"Wallet RPC won't connect to daemon"
  - Ensure daemon 1 is running on port 34568
  - The GUI tool's wallet-rpc uses --daemon-ssl=disabled; make sure you
    haven't reverted that change in electron/main.js

"No balance showing in wallet"
  - Coinbase outputs need confirmations to unlock (5 blocks after HF8).
    Ensure the chain is well past height 150 before checking balances.
  - Open wallet in CLI and type 'refresh' then 'balance'.

"Migration transaction fails"
  - The chain must be past HF11 (height 60) for v3 migration transactions.
  - The testnet portal address keys differ from mainnet — this is handled
    automatically when using --testnet.

"Failed to create block template"
  - If V9_BLOCK_HEIGHT in cryptonote_basic_impl.cpp wasn't changed to 40,
    get_block_reward() will underflow and fail. Rebuild after the fix.
